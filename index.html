⚠ Migration Risk

PD teams working from the new templates are producing content that maps to fields not yet in Coda — while the existing Coda doc still expects the old schema. Programs must not be asked to enter new-template data into old Coda docs. Any program transitioning to these new templates needs a fresh, scaffolded Coda doc built against the new schema before data entry begins.

Architect Note — Highest Priority: The GE Outcomes mapping is the most significant structural gap. The current Coda schema has _Progs Outcomes with an Outcome Type field ("Program Outcome" vs "GE Outcome") but the import pipeline has never populated GE Outcomes — and the junction tables for alignment have no GE rows. Options: (A) Create two new junction tables (Course→GE, Comp→GE) mirroring the existing Program Outcome junction tables, or (B) Extend the existing junction tables with a type filter. Option A is cleaner for querying. The "A-only" constraint for GE at course level (vs full IRMA for program outcomes) also means the alignment select list logic must be conditioned by outcome type.
Recommendation

Clean Coda Docs for New-Template Programs

The Problem

The existing Coda doc ("Design & Development V3") was built for the old template schema. The new templates add fields, rename tables, and restructure content in ways that are not backward compatible. A PD team working from the new CCW, V&S, or PDOW templates will produce content that:

  • Has no home in current Coda columns (e.g., Course Rationale/Vision, Portfolio Crosswalk, GE Outcome mappings)
  • Maps to renamed concepts that may create confusion in reporting (e.g., "Learning Experience" vs. "Learning Resource")
  • Omits fields Coda currently requires or expects (e.g., ASMT Notes, Course Modality)

Asking a PD team to enter new-template data into the old Coda doc would either force them to ignore new fields, shoehorn content into wrong fields, or produce data that is inconsistent with every other program in that doc — all of which would degrade data integrity and create rework later.

Recommendation: New Coda Doc per Program on New Templates

Any program transitioning to the new templates should be set up in a new Coda doc built to the new schema — not entered into the existing V3 doc. Specifically:

  1. Freeze the existing V3 doc for programs already in it. Do not add new programs to V3 once the new-schema doc exists. V3 becomes "legacy" but remains the source of truth for programs already active there.
  2. Build a "Design & Development V4" (or equivalent) with updated tables, new columns, and renamed strategy tables. This doc should be scaffolded and ready for a program before their PD team begins entering any data.
  3. Establish a clear cutover gate: a program gets entered into V4 only after: (a) PMO confirms they are using the new templates, and (b) the Coda architect confirms V4 scaffolding is in place for that program.
  4. Do not attempt an in-place migration of V3 → V4 schema. The column additions and renames are significant enough that a clean new doc reduces risk substantially vs. modifying live production tables.

Open Questions Requiring PMO Decision

  • Practicum / Clinical tables: Removed from base V&S template — are they still needed for health programs? If yes, the new doc needs optional sub-tables for these.
  • Eco-System Components row: Should this select-list value be retired from the Ecosystem table, or preserved for programs that may still use it?
  • Portfolio Crosswalk: Is this intended to be Coda-native data, or is it a reference document that lives only in the V&S DOCX? If Coda-native, a new table design is needed before any import work begins.
  • [Additional Feature] Strategy: How many program-specific named features might appear? This affects whether to design a typed/flexible table vs. separate tables per feature type.
  • CCW Scope/Redev flags: These moved to the PDOW. Does the Coda workflow need to ingest them from there, or will they be entered manually?
Guidance

How to Present This to Each Audience

This analysis serves three different audiences with different needs. Presenting the same table to all three will either over-inform or under-inform.

🎯 For Kristen & PD PMO

  • Lead with the risk banner — frame it as a data integrity risk, not a technical problem.
  • Focus on: which templates changed, what PD teams will notice, and the "no new programs in V3" rule.
  • Present the Open Questions list and request decisions on each item.
  • Suggest a "go/no-go" gate: program gets a Coda doc only after PMO confirms template version.
  • Format: 1-page summary + this doc as a reference appendix. Or a short slide deck (5 slides: Risk, CCW changes, V&S changes, PDOW changes, Recommendation).

🏗 For the Coda Architect

  • The full diffs table is the primary deliverable — especially the purple "Architect Note" callouts.
  • Key decision points to surface: GE Outcome junction tables, Portfolio Crosswalk table design, flexible Additional Feature strategy table, and column deprecation strategy for old CCW fields.
  • Request an estimate for V4 scaffolding time before any program is promised a date.
  • Format: This HTML document as-is. Add a schema design review meeting.

📊 For PD Leadership & Management

  • Focus on the recommendation section only — they don't need the detailed diffs.
  • Frame it as a system readiness issue: the tool must be ready before teams start.
  • Highlight the clean-doc recommendation as a risk mitigation investment, not overhead.
  • Give a concrete ask: approve the V4 doc build before the first new-template program kicks off.
  • Format: 2-paragraph summary email with the recommendation section attached.

📋 Combined Review Meeting

  • If presenting to all three groups together, use this document as the agenda backbone.
  • Walk the diffs tables briefly (10 min) — most questions will cluster around CCW restructuring and PDOW GE Outcomes.
  • Spend the majority of time (20–30 min) on the Open Questions and getting PMO decisions on record.
  • End with a clear action list: who owns V4 scaffolding, when the cutover gate activates, and who communicates to PD teams.
